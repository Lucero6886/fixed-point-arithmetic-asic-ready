# Project 8.3.1: Yosys Synthesis Study For SC Decoder N=16 Resource-Shared RTL

## 1. Project Objective

Project 8.3.1 performs Yosys synthesis for the resource-shared scheduled SC Decoder N=16 developed in Project 8.3.

The target RTL file is:

```text
rtl/sc_decoder_n16_shared.v
```

The top module is:

```text
sc_decoder_n16_shared
```

The main objective is to obtain a synthesis-level complexity baseline for the functionally verified multi-cycle resource-shared N=16 decoder.

Project 8.3 already verified that the shared decoder passes:

```text
1000 N=16 golden vectors
0 errors
115-cycle deterministic latency
ALL TESTS PASSED
```

Project 8.3.1 now answers the following question:

```text
How large is the resource-shared scheduled SC Decoder N=16 after flattened Yosys synthesis?
```

This project is not a new RTL design project.

It is a synthesis characterization project for the verified resource-shared N=16 RTL.

---

## 2. Why Project 8.3.1 Is Important

The resource-shared N=16 decoder is already functionally correct.

However, functional correctness alone does not prove architectural benefit.

A resource-shared architecture is meaningful only if it improves some hardware metrics compared with the reference baseline.

Therefore, Project 8.3.1 is needed to measure:

```text
number of wires
number of wire bits
total cell count
DFF/DFFE cell count
estimated combinational cell count
MUX cell count
XOR/XNOR cell count
NAND cell count
AND/OR cell count
```

The key research question is:

```text
Does the resource-shared N=16 RTL reduce duplicated combinational logic compared with the N=16 reference RTL?
```

This requires comparison between:

```text
Project 8.2.1:
    SC Decoder N=16 reference RTL Yosys synthesis

Project 8.3.1:
    SC Decoder N=16 resource-shared RTL Yosys synthesis
```

Without this project, the resource-shared N=16 design would only be functionally correct, but not yet quantitatively characterized.

## 7. Yosys Script

The synthesis script should be:

```text
synth/sc_decoder_n16_shared_flat.ys
```

The intended Yosys flow is:

```tcl
read_verilog -sv rtl/sc_decoder_n16_shared.v

hierarchy -check -top sc_decoder_n16_shared

proc
opt
flatten
opt

techmap
opt

abc
opt
clean

stat
```

The key commands are:

```text
flatten
abc
stat
```

Flattening is used to expose the full primitive-cell-level complexity.

ABC mapping is used to map the design into generic gate-level cells.

The final `stat` report provides the metrics extracted by the Python summary script.

## 8. Why Flattened Synthesis Is Used

The generated resource-shared RTL is structurally large and contains many named registers, control states, and datapath expressions.

Flattened synthesis answers:

```text
How many primitive logic cells are inferred after expanding the complete resource-shared decoder?
```

This is useful because Project 8.4 must compare two complete N=16 designs:

```text
sc_decoder_n16_ref
sc_decoder_n16_shared
```

A fair comparison should use the full expanded design, not only module instances.

Therefore, flattened Yosys synthesis is appropriate for Project 8.3.1.

## 9. Sequential Logic Expectation

Unlike the N=16 reference RTL, the resource-shared decoder is sequential.

Therefore, DFF/DFFE cells are expected.

These sequential cells come from:

```text
FSM state register
input LLR registers
intermediate LLR registers
partial-sum registers
decoded-bit registers
frozen-mask register
u_hat output register
busy register
done register
```

Therefore, the resource-shared design should not be judged only by total cell count.

The meaningful metrics are:

```text
total cells
DFF/DFFE cells
estimated combinational cells
latency cycles
```

The estimated combinational cells are computed as:

```text
estimated_comb_cells = total_cells - DFF/DFFE_cells
```

This separates datapath/control combinational complexity from sequential storage overhead.

## 11. Resource-Shared Architecture Under Synthesis

The design synthesized in this project is a multi-cycle resource-shared scheduled SC Decoder N=16.

It uses:

```text
start/busy/done interface
FSM controller
one shared f/g datapath
internal LLR registers
partial-sum registers
decoded-bit registers
u_hat output register
schedule-guided operation sequence
```

The measured functional latency is:

```text
latency_cycles = 115
```

This latency should be included in future architecture comparison.

The design trades:

```text
more cycles
more registers
more FSM/control logic
```

for:

```text
less duplicated combinational f/g datapath logic
shorter per-cycle combinational path potential
better scalability toward larger N
```

## 17. Expected Architecture Trade-Off

The expected trade-off is:

```text
N=16 reference RTL:
    purely combinational
    no DFF/DFFE cells
    larger duplicated combinational logic
    very long combinational path expected

N=16 resource-shared RTL:
    sequential multi-cycle design
    many DFF/DFFE cells
    less duplicated combinational logic expected
    115-cycle deterministic latency
```

Therefore, the correct interpretation is not:

```text
resource-shared is always smaller in total cells
```

The more careful interpretation is:

```text
resource-shared should reduce estimated combinational cells and critical-path potential,
but introduces sequential storage and multi-cycle latency.
```

This distinction is important for academic writing.

## 18. Why Total Cell Count Alone Is Not Enough

The resource-shared design contains many registers.

Therefore, total cell count includes both:

```text
combinational cells
sequential cells
```

The reference RTL is expected to be mostly or entirely combinational.

Therefore, comparing total cells alone may be misleading.

A fairer comparison should report:

```text
total cells
DFF/DFFE cells
estimated combinational cells
latency cycles
```

Example interpretation pattern:

```text
Although the resource-shared design introduces sequential cells for state and storage,
it reduces estimated combinational complexity compared with the reference decoder.
```

This statement should only be used after actual Yosys results confirm it.

---

## 19. Latency-Aware Interpretation

Project 8.3 measured:

```text
latency_cycles = 115
```

Therefore, any architecture comparison must include latency.

The effective decoding time should later be computed as:

```text
decode_time = latency_cycles × clock_period
```

For the resource-shared decoder:

```text
decode_time_shared = 115 × clock_period_shared
```

For the reference combinational decoder, the latency is conceptually one combinational evaluation, but it may require a much longer clock period if registered as a one-cycle design.

Therefore, future OpenLane comparison should consider:

```text
clock period
latency cycles
effective decode time
area
area-latency product
```

## 22. Common Problems And Debugging

### Problem 1: Yosys Cannot Find The Top Module

Possible cause:

```text
wrong top module name
```

Correct top module:

```text
sc_decoder_n16_shared
```

Check with:

```bash
grep -n "module sc_decoder_n16_shared" rtl/sc_decoder_n16_shared.v
```

The Yosys script should contain:

```tcl
hierarchy -check -top sc_decoder_n16_shared
```

---

### Problem 2: RTL Was Not Regenerated

Because the RTL is generated from a schedule, stale RTL can become a problem.

The run script should regenerate:

```text
results/schedules/sc_decoder_n16_schedule.csv
rtl/sc_decoder_n16_shared.v
```

before running Yosys.

Recommended commands:

```bash
python3 model/sc_schedule_generator.py
python3 scripts/generate_sc_decoder_n16_shared_rtl.py
```

---

### Problem 3: Extract Script Cannot Find The Summary Block

Possible cause:

```text
Yosys failed before stat
or
top module name is different
```

Check:

```bash
grep -n "=== sc_decoder_n16_shared ===" synth/reports/sc_decoder_n16_shared_flat_yosys.log
```

---

### Problem 4: DFF Count Is Nonzero

This is expected.

The resource-shared decoder is sequential.

DFF/DFFE cells are expected due to:

```text
FSM state
LLR registers
decoded-bit registers
partial-sum registers
u_hat register
busy/done registers
```

Nonzero DFF/DFFE count is not a bug.

---

### Problem 5: Total Cell Count Seems Large

A large total cell count may still be reasonable because the design includes many registers and control signals.

Use:

```text
estimated_comb_cells = total_cells - DFF/DFFE_cells
```

for a more meaningful datapath/control comparison.
